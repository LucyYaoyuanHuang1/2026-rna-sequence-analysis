Lesson 1
* A cluster is a collection of compute nodes that you access from your terminal
* A computer node is a single unit of a computer
* A login node is a shared access point for all users
o Do NOT run computational tasks on the login nodes
o Points to $HOME, $SCRATCH, and compute nodes
* You will login to a login node
* Then create a file that has instructions to the scheduler
o Eg. X memory, x CPU cores, x time, x command
* This file is sent to the scheduler which runs the job on the compute node that it selects
* Batch system run through a scheduler
o Batch means the program running is disconnected from the terminal (aka your computer)
* $HOME is home, has storage you may compute against
* $SCRATCH has additional storage you may compute against
* DO NOT run computation tasks on the login node
o Allowed: download and compile items
o NOT allowed: anything computational
* Seadragon cluster example
o Top: high-speed network InfiniBand, also has ethernet
o Middle: management nodes and some login nodes
o Bottom: 24 compute nodes

* Submitting jobs
o Scheduler: how you submit jobs to the cluster
o Fair share: jobs are assigned priority based on active usage
* Less-active users have higher priority
o Job limits: total CPUs requested across all jobs, per queue, jobs will be put on pending until under limit and then more will be submitted
* Total memory  requested across all jobs, per queue, large limit no problem
o GPU queue: 10 CPU cores are automatically assigned for every GPU core requested
* Please use the least amount of memory and wall time (time based on the clock on the wall) needed for each job
o Common queues
* 3 queues – Cdragon, Edragon (e40 and e80), (28 core, 165/350g), (40 core, 475G), (80 core, 950G)
* Split into short, medium, and long based on wall time (3min, 1 day, 10 days respectively
* Eg. Short medium long e40short e40medium e40long e80short e80medium e80long
o Specialty queues
* Interactive queue: command line access on a compute node, quickly test code without running on login node
* Eint: also interactive, but for jobs on edragon long
* Transfer: transfering data between storage devices, accessing data from external data sites such as EGA and GDC, only queue that can access internet
* Gpu: gpu nodes
* Highmem: accessing nodes with multiple terabytes of memory
* Emultinode: jobs that run on multiple nodes, special software required, invitation only (contact HPC-Team)

Lesson 3
* Submitting jobs
o Complete examples with instructions found at /risapps/example-scripts
o In linux, command line is called bash, so script is bash script
o Example script
* #BSUB  -W 24:00
* #BSUB – aka Batch SUBmit, aka tells scheduler “this is for you”
* -W 24:00 – wall time 24:00 request
* #BSUB –q medium
* Medium queue
* #BSUB –o /rsrch3/scratch/your/logdir/or_file
* Output to /rsrch3/scratch/your/logdir/or_file
* Recommended to create log directory for my own project, in scratch
* #BSUB –cwd /rsrch3/scratch/you/working/directory
* Set working directory /rsrch3/scratch/you/working/directory
* Can be in home or scratch
* #BSUB –u myemail@mdanderson.org
* Sets email, if job fails you will receive email
* #BSUB –n 12
* 12 cores
* #BSUB –M 16
* Request 16 GB memory
* #BSUB –R rusage[mem=16]
* Reserve 16 GB
* -M and –R must be the same
* #BSUB –J jobNameHere
* Optional
* Upper, lower, special, underscore, etc but no spaces
* #BSUB –P projectNameHere
* Optional
* Get project info from PI
* Module load app_module/1.2.3
* Tells system where to find application you are running
* Make sure application is in working directory if using own software
* Else make sure to put full path
* Myprogram arg1 arg2 –threads=12 \
* If multiple threads needed
* Must set to same number as number of cores
* Input=/rsrch3/home/your/project/data\
* Output=/rsrch3/scratch/your/project/out
* Input and output can be both home or scratch
o Managing jobs
* Seadragon uses the LSF job scheduling script from IBM
* Bsub < myjob.lsf
* Submits myjob.lsf (which is found in the current directory)
* Reply: Job <xxxxxx> is submitted to queue <medium>
* Type into terminal
* Bjobs 
* To view jobs in queue
* -p for pending, use if your job isn’t submitted after a while
* -u all for to view all jobs in queue owned by all users
o EXEC_HOST 5*xxxxx means 5 cores
o JOB_NAME *xxxx means name cut off
o 
* Bkill JobID
* To cancel job
o Interactive jobs
* Can use both submit script and command line
* [user@ldragon2] $ Bsub –Is –q interactive –W 6:00 –M 16 –T rusage[mem=16] -n1 /bin/bash
* -Is means interactive (may be submitted to any queue
* \ escapes carriage return
* 1 core is typically enough
*  output: Job <xxxxxx> is submitted to queue <interacti>.
o <<waiting for dispatch...>>
o <<starting on cdragon304>>
* [user@cdragon304] $ (note changed from red to green)
* color change denotes change from login node to compute node
o Useful commands
* bjobs –uall | more
* show all jobs in queue
* bjobs –l xxxxxx
* show detail on one job
* bjobs –p
* show pending reasons
* bkill xxxxxx
* cancel submitted job
* bqueues
* see queue status (slots)
* bqueues-jobs.sh
* see queue usage (jobs/slots)
* bhosts
* see host status
* can provide with list of host names
* lsload –gpuload
* load gpu load


* can provide with list of host names
* bmod –W xx:xx
* replace wall time

Lesson 5
* Environment modules
o modules manage all of the shared software on the cluster
o module avail packageName - see which software packages are available
o module load moduleName – load specific module
o module unload moduleName
o show, help, list (list all loaded modules), purge (clear all modules) – other helpful commands
o variables are automatically carried forward into compute jobs, even when typed on command line
o (D) means default, (L) means loaded

* all containers on cdragon use a different engine from Docker
* Containers are software used to encapsulate an enviroment

o /temp is small, be careful
* to run a Singularity container on gpu, use –nv
